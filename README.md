# 構成要件推理示範系統

一個簡化版的法律要件推理示範網頁,重現某博士論文第五章「MECE驅動法律知識圖譜與混合神經符號推理」三階段管線的核心邏輯,供論文口試討論之用。**非正式法律工具,結果僅供教學與方法論討論參考。**

## 這是什麼

輸入一段中文案件事實,系統會依序執行:

1. **要件抽取(呼叫 OpenAI API)**:針對13個預先定義的構成要件,由語言模型逐項判斷是否成立、給出信心分數與判斷依據。
2. **符號邏輯核對(純本機運算,不呼叫 AI)**:六條法條(§320竊盜、§321加重竊盜、§335侵占、§339詐欺、§339-2 ATM詐欺、§339-4加重詐欺)的 AND/OR 構成要件規則,寫死在程式碼中,直接比對第一步的結果。
3. **判定**:輸出符合的法條;若多條法條同時符合,明確標示「衝突,需人工判斷」,不自動猜一個答案。

## 為什麼這樣設計

這個demo刻意保留了論文方法論裡值得討論的兩個特徵,方便口試時實際操作觀察:

- **要件抽取這一步,語言模型是一次性判斷,沒有二次人工覆核**——對應論文§5.6.3「LLM精煉」步驟的設計方式與其潛在風險。
- **§339-2(ATM詐欺)的規則,刻意只用「對機器施用不正方法」建構,不包含「欺騙自然人」的要件**,與§320、§339-4的規則結構明確分開,不套用「上位/特定概念」的階層關係——用意是對照示範:知識圖譜的規則若設計得當,就不會出現論文中「將§339-2誤建模為§320或§339-4之特殊版本」的問題。

## 使用方式

1. 下載本資料夾(`index.html`),用瀏覽器直接開啟即可,或部署到 GitHub Pages。
2. 準備一組 [OpenAI API 金鑰](https://platform.openai.com/api-keys)。
3. 開啟頁面後,於金鑰欄位貼上金鑰(隱碼輸入),選擇模型(建議先用 `gpt-4o-mini` 測試,較快也較便宜)。
4. 輸入案件事實,或點選預設範例按鈕(其中「論文爭議例句」對應論文中討論過的模糊案例)。
5. 按「開始分析」,依序查看三個階段的輸出。

## 部署到 GitHub Pages

```bash
git init
git add index.html README.md
git commit -m "init"
git branch -M main
git remote add origin <你的repo網址>
git push -u origin main
```

推送後至 Repo 的 **Settings → Pages**,Source 選擇 `main` 分支、`/ (root)` 資料夾,儲存後幾分鐘內即可透過 GitHub 提供的網址開啟。

## ⚠️ 安全性提醒

- 本頁為純前端(無後端伺服器),API 金鑰會**直接從使用者瀏覽器**發送至 OpenAI,不經任何中介伺服器,也不會被寫入 GitHub Pages 或任何雲端儲存。
- 金鑰僅存於當前分頁的瀏覽器記憶體中,重新整理頁面即清除,頁面本身**不使用** localStorage / cookie 保存金鑰。
- 這樣的設計適合**個人測試、口試現場示範**等場景。若之後打算公開此網址供他人(例如口試委員、同學)自行輸入金鑰使用,建議先評估金鑰外洩風險,或改為後端代理呼叫 API 的架構。
- 請勿將含有你個人金鑰的瀏覽器分頁畫面截圖或錄影分享出去。

## 範圍與限制

- 僅涵蓋竊盜/侵占/詐欺家族的**六條法條**、**13個構成要件**,是刻意縮小範圍的教學示範版,不是完整重現論文17條法條、646條關係的正式系統。
- 要件抽取結果的品質,取決於所選用的 GPT 模型與 prompt 設計,可能與論文原始使用的 Gemini 2.0 Flash 表現不同,僅供方法論層次的討論參考,不代表論文原系統的實際輸出。
- 本系統不做任何案件事實以外的法律諮詢或建議。
